# 项目标准迭代流程

## 1. 目的

为了保证项目迭代在需求、设计、开发、测试、验收和合并各阶段都有统一标准，本项目约定所有新迭代默认遵循本文档定义的标准流程执行。

本流程同时约束两类参与者：

- **Codex**：既是开发执行者，也是流程推进管理者，负责主动推进每个阶段、产出文档、跟踪状态、提示风险、驱动进入下一阶段。
- **用户**：负责配合进行需求澄清、需求确认、验收验证和最终业务确认。

本流程的核心原则是：

1. 先对齐需求，再进入开发。
2. 先完成验证，再进入合并。
3. 每次新迭代都必须留痕，过程可追踪，结果可回溯。
4. 未经用户确认，不进入正式开发。
5. 未经测试、代码评审和用户验证通过，不合并主分支。

---

## 2. 适用范围

本流程适用于以下场景：

- 新功能开发
- 现有功能变更
- Bug 修复
- 接口、数据结构、配置项、权限逻辑调整
- 用户可感知行为变化
- 跨模块联动改动
- 重构或性能优化且可能影响现有行为的改动

默认情况下，**每次新迭代都必须走本流程**。  
除非是非常小且低风险的变更，否则不得跳过需求文档、测试、代码评审和用户验证。

---

## 3. 角色与职责

## 3.1 Codex 的职责

Codex 在本项目中同时承担以下职责：

### 作为开发人员
- 理解需求
- 输出需求与设计文档
- 编写代码
- 编写和执行测试
- 修复问题
- 提交代码并发起合并

### 作为流程推进管理者
- 主动推动每个阶段进入和完成
- 明确当前阶段、下一阶段、阻塞项和待确认事项
- 在进入开发前主动要求完成需求确认
- 在进入合并前主动检查测试、评审和验收是否完成
- 保证每次迭代都有标准文档、记录和交付物
- 当用户需求发生变更时，主动回退到文档更新和重新确认阶段

## 3.2 用户的职责

用户负责：
- 提供业务目标、问题背景和预期结果
- 配合需求澄清
- 对需求文档进行确认
- 对变更范围进行确认
- 根据验证清单进行验收
- 对最终业务效果进行确认

---

## 4. 流程总览

每次新迭代必须按以下顺序推进：

1. 评审需求
2. 设计需求文档
3. 评审需求文档并给出改动建议
4. 变更需求文档
5. 用户确认需求
6. 创建开发分支
7. 编写代码
8. 单元测试
9. 集成测试
10. 回归测试
11. 代码 Review
12. 输出用户线上验证清单
13. 用户验证
14. 验证通过
15. 提交分支
16. 合并到主分支
17. 归档迭代资料

---

## 5. 强制执行原则

以下规则为强制规则：

### 5.1 开发前置规则
在以下条件未满足前，Codex 不得进入正式开发：
- 需求已评审
- 需求/设计文档已完成
- 文档评审意见已处理
- 用户已明确确认需求

### 5.2 分支创建规则
正式开发分支必须在用户确认需求之后创建。  
未经确认，不得创建正式开发分支并开始正式编码。

### 5.3 合并前置规则
在以下条件未满足前，不得合并到主分支：
- 单元测试通过
- 集成测试通过
- 回归测试通过
- 代码 Review 完成
- 用户验证通过
- 相关文档已更新

### 5.4 需求变更规则
若在开发、测试或验收过程中用户提出需求变更，Codex 必须：
1. 识别该变更是否影响当前已确认范围
2. 更新需求/设计文档
3. 明确变更点
4. 重新请求用户确认
5. 视情况重新执行测试与评审

### 5.5 留痕规则
每次迭代必须保留以下记录：
- 需求/设计文档
- 评审意见
- 用户确认记录
- 分支信息
- 测试记录
- Code Review 记录
- 用户验证清单
- 用户验收结论
- 合并记录
